## Design

### Redis — managed ElastiCache 7

Uses `aws_elasticache_replication_group` (modern API) in **cluster-
mode-disabled** with a single node group. Topology toggles via
`replicas_per_node_group`:

| `replicas_per_node_group` | Topology | HA |
|---|---|---|
| `0` | single primary | none — accepts data loss on AZ failure (staging) |
| `>= 1` | primary + N replicas | `multi_az_enabled = true`, `automatic_failover_enabled = true` (production) |

The HA toggles are derived (`var.replicas_per_node_group > 0`), not
exposed as separate booleans — preventing the misconfiguration where
an operator enables multi-AZ without replicas (which AWS rejects).

**Encryption** — `transit_encryption_enabled = true` is **hardcoded**
(OI-1 spec), so the structural ratchet asserts no `var.` indirection
on it. `at_rest_encryption_enabled = true` likewise. KMS key is
optional via `var.kms_key_arn` (default null = AWS-managed).

**AUTH token** — generated by `random_password` (length 64,
`override_special = "!&#$^<>-"` to match ElastiCache's allowed
character set), wired via `auth_token = random_password.auth.result`.
Stored in AWS Secrets Manager as a self-describing JSON blob (same
shape pattern as the RDS-managed master credentials secret). The app
reads this at runtime and builds:

```
REDIS_URL = rediss://<auth_token>@<primary_endpoint>:6379
```

The `rediss://` scheme is what `ioredis` uses for TLS. The existing
app code at `src/lib/redis.ts` accepts this transparently — no
client-side change needed.

**Parameter group** on the `redis7` family. Single non-default value:

| Parameter | Value | Why |
|---|---|---|
| `maxmemory-policy` | `noeviction` | **BullMQ requirement.** The default `volatile-lru` would evict TTL-bearing keys under memory pressure, silently losing job state. `noeviction` makes the cache push back on the client (returns `OOM` error) instead of dropping data. |

**Networking** — `aws_elasticache_subnet_group` over the **private-app**
subnet tier (Redis lives next to the app workload, not in the DB
tier). Security group is created in this module with **exactly one
ingress rule** referencing `var.app_security_group_id` — no CIDR
ingress anywhere; ratchet-asserted.

**Snapshots** — `snapshot_retention_days` validated `>= 1` (PITR-
equivalent always on). Staging defaults to 1; production extends to 7.

**Log delivery** — slow-log + engine-log shipped to CloudWatch when
`enable_log_delivery = true` (default). Two log groups created
in-module.

### Storage — S3 with IA lifecycle + IAM-driven access

Standard production-grade S3 setup:

| Control | State |
|---|---|
| Versioning | **Enabled** (default; OI-1 spec). Status wired to `var.versioning_enabled`. |
| SSE | **AES256 (SSE-S3)** — explicitly *not* SSE-KMS (OI-1 spec). No `kms_master_key_id` in the encryption block; ratchet-asserted. |
| Public-access-block | All four flags `true` (block public ACLs, block public policy, ignore public ACLs, restrict public buckets). Ratchet-asserted. |
| Bucket policy | Single statement: deny `s3:*` when `aws:SecureTransport = false` (no plaintext access ever, even from inside AWS). |
| Lifecycle | (a) current → `STANDARD_IA` after 90 days (OI-1 spec); (b) noncurrent versions expire after 365 days; (c) incomplete multipart uploads aborted after 7 days. |

**CORS** — only when `cors_allowed_origins` is non-empty:

```
allowed_methods = ["PUT", "POST", "GET", "HEAD"]   # PUT for pre-signed URL upload
allowed_origins = var.cors_allowed_origins         # explicit list, no wildcard
allowed_headers = ["*"]                            # pre-signed PUT forwards Content-Type, x-amz-meta-*
expose_headers  = ["ETag"]                         # client-side upload integrity check
max_age_seconds = 3600
```

The `aws_s3_bucket_cors_configuration` resource is `count = local.has_cors ? 1 : 0` — server-side-upload-only deployments get no CORS rule at all (default empty list). Wildcard origin (`*`) is structurally
disallowed: env tfvars must specify explicit origins, and the ratchet
asserts no `"*"` appears in either env's CORS list.

**Why CORS matters here.** The app uses pre-signed URLs (`getSignedUrl`
from `@aws-sdk/s3-request-presigner`) — the **browser** does the PUT
directly to S3. Without CORS, the browser preflight fails. Confirmed
by inspection of `src/lib/storage/s3-provider.ts` which calls
`getSignedUrl` against `PutObjectCommand` and `GetObjectCommand`.

**IAM-driven access (not bucket-policy grants).** The module creates
a single `aws_iam_policy` named `<name_prefix>-storage-access` with:

```
ListBucket            on  arn:aws:s3:::<bucket>
GetBucketLocation     on  arn:aws:s3:::<bucket>
GetObject             on  arn:aws:s3:::<bucket>/*
GetObjectVersion      on  arn:aws:s3:::<bucket>/*
PutObject             on  arn:aws:s3:::<bucket>/*
DeleteObject          on  arn:aws:s3:::<bucket>/*
DeleteObjectVersion   on  arn:aws:s3:::<bucket>/*
AbortMultipartUpload  on  arn:aws:s3:::<bucket>/*
ListMultipartUploadParts on arn:aws:s3:::<bucket>/*
```

The policy is **always created** (so the ARN is a stable contract).
The role is **optionally created** via `create_app_role = true` — when
ECS lands in a future epic, the operator passes
`app_role_assume_principals = ["ecs-tasks.amazonaws.com"]`. For now
the policy ARN is exposed as a root output for attachment to whatever
workload role exists. Ratchet asserts the policy resource has no
`count` (always created) and the role resource does have `count` (gated).

### Staging vs production tradeoffs

| Knob | Staging | Production |
|---|---|---|
| `redis_node_type` | `cache.t4g.small` | `cache.t4g.medium` |
| `redis_replicas_per_node_group` | 0 (single-node) | 1 (HA + multi-AZ + automatic failover) |
| `redis_snapshot_retention_days` | 1 | 7 |
| `storage_cors_allowed_origins` | `["https://staging.example.com"]` | `["https://app.example.com"]` |
| `storage_force_destroy` | `true` (allow teardown) | `false` (NEVER destroy prod bucket) |

## Decisions

- **`transit_encryption_enabled = true` hardcoded.** OI-1 spec
  requires encryption-in-transit. Putting it behind a `var.` would
  let an operator silently disable it; structural ratchet asserts
  no `var.` indirection.

- **`maxmemory-policy = noeviction`, not `allkeys-lru`.** BullMQ
  stores job state in Redis. Under memory pressure, eviction means
  job records disappear and workers can't update or complete them.
  The right behaviour is push-back to the client (`OOM` error) so
  the system degrades visibly instead of silently losing work. This
  is a load-bearing operational invariant; the parameter group
  enforces it cluster-wide.

- **AUTH token via `random_password` + Secrets Manager, not
  ElastiCache user groups.** ElastiCache user groups are the modern
  AUTH primitive but require Redis 6+ ACL setup that the app driver
  (`ioredis` configured to send `AUTH <token>`) doesn't currently
  support without changes. The legacy `auth_token` field gives us
  the same TLS+token coverage with zero app changes.

- **Redis lives in the private-app subnet tier, not private-db.**
  The app constantly reads/writes to Redis; co-locating it in the
  app tier avoids cross-tier traffic. The DB tier stays maximally
  isolated (no internet egress, fewer SG exceptions).

- **HA toggles derived from replica count, not separate booleans.**
  AWS rejects `multi_az_enabled = true` without replicas, and rejects
  `automatic_failover_enabled = true` without replicas. Computing
  both from `var.replicas_per_node_group > 0` makes the
  misconfiguration impossible.

- **SSE-S3 (AES256) over SSE-KMS.** OI-1 spec is explicit. SSE-KMS
  would require kms:Decrypt + kms:GenerateDataKey on the workload
  role, plus a per-bucket key; SSE-S3 is sufficient for the
  compliance bar OI-1 sets, and avoids the extra IAM surface.
  Migration to SSE-KMS later is a one-line change in `main.tf` (and
  a bump to the workload-policy template).

- **Public-access-block before bucket policy attachment.** AWS
  rejects bucket policy attaches that "could be" public until the
  block is in place. The `depends_on = [aws_s3_bucket_public_access_block.this]`
  on `aws_s3_bucket_policy.this` enforces the order so a clean apply
  works on first run.

- **CORS gated on `length(cors_allowed_origins) > 0`.** Two reasons:
  (a) deployments that don't use pre-signed URL uploads get no CORS
  rule, which is one fewer attack-surface knob. (b) An empty list
  with the resource always-on would create an empty `allowed_origins`
  which AWS rejects. The structural ratchet asserts the count
  expression is `local.has_cors ? 1 : 0`.

- **IAM policy always created; role optional.** The policy ARN is a
  stable contract every workload-role consumer can attach. Creating
  the role optionally avoids locking the storage module to a single
  compute layer (EC2 instance profile vs ECS task role vs Lambda
  exec role) — the operator picks the right principal for their
  stack.

- **No wildcard origin in CORS.** The ratchet asserts each env's
  `storage_cors_allowed_origins` list contains explicit `https://`
  origins and no `"*"`. A future PR adding `"*"` for "convenience"
  fails CI in the same diff.

- **`force_destroy = false` default.** Tenant evidence files are
  irreplaceable. The structural ratchet doesn't lock this default
  (operators in dev sandboxes legitimately want true), but the
  per-env policy is encoded: production tfvars sets `false`
  explicitly with a comment.

## Verification performed

- **Structural ratchet**: 97/97 green across all three terraform
  guard tests (40 foundation + 26 VPC/DB + 31 Redis/Storage).
  Specifically locks: Redis TLS-in-transit hardcoded; AUTH wired
  from `random_password` to Secrets Manager; Redis ingress only from
  app SG (no CIDR); `redis7` family with `maxmemory-policy = noeviction`;
  HA toggles derive from replica count; cluster-mode-disabled;
  storage public-access-block all-on; versioning default true; SSE
  = AES256 (no KMS); IA-90d transition; CORS gated on non-empty
  origins (count = local.has_cors ? 1 : 0); deny-non-TLS bucket
  policy; IAM policy always created (no count); IAM role gated on
  `var.create_app_role`; bucket policy contains no `Allow` effect
  (only deny-non-TLS); per-env tradeoff overrides match policy;
  CORS origins have no wildcard.
- **Brace-balance** across all 9 module + root + env files: clean.
- **Module file sizes**: redis 387 lines (main + vars + outputs),
  storage 418 lines. Both well under the "single concern per
  module" smell threshold.
- **`terraform fmt` / `terraform init` / `terraform validate`**: not
  run — the terraform binary isn't installed in this sandbox. CI
  wiring on `infra/terraform/**` is the follow-up; the 97-assertion
  ratchet is the day-one substitute.
- **Cross-module integration**: ratchet asserts `module.redis`
  consumes `module.vpc.private_app_subnet_ids` (correct tier) and
  `module.vpc.app_security_group_id` (correct SG). Storage module
  has no VPC dependency (S3 is global).
