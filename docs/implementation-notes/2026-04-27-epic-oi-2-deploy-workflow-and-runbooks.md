## Design

### Deploy workflow rewrite

Old `.github/workflows/deploy.yml`:
- Trigger: `workflow_dispatch` (manual, env + ref inputs)
- Build image → SSH into the deploy host → `docker compose pull && docker compose up -d` → smoke test
- Required secrets per env: `DEPLOY_HOST`, `DEPLOY_USER`, `DEPLOY_SSH_KEY`, `DEPLOY_PATH`

New deploy.yml (this PR):
- Trigger: `workflow_dispatch` (same shape, plus an explicit `image_tag` input)
- Build image → OIDC into AWS → `aws eks update-kubeconfig` → `helm lint` → `helm upgrade --dry-run` → `helm upgrade --install --atomic --wait --timeout 10m` → smoke test
- Required secrets per env: `AWS_ROLE_TO_ASSUME` (already used by terraform.yml), `EKS_CLUSTER_NAME` (new)
- Required vars per env: `SMOKE_URL` (was hardcoded; now per-env GH Environment variable)
- All four legacy `DEPLOY_*` SSH secrets are **removed** from the workflow

### Job pipeline

```
workflow_dispatch (operator chooses env + ref + optional image_tag)
       │
       ▼
┌─────────────┐
│   gate      │  resolves env, ref, image_tag (short-SHA fallback),
│             │  release name (inflect-<env>), namespace (inflect-<env>)
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ build-image │  builds + pushes ghcr.io/<repo>:<tag>
└──────┬──────┘
       │
       ▼
┌──────────────────────────────────────────────┐
│ deploy   environment: <staging|production>   │  ← required-reviewers gate on prod
│  ├── OIDC AWS auth                           │
│  ├── aws eks update-kubeconfig               │
│  ├── helm lint                               │
│  ├── helm upgrade --install --dry-run        │
│  ├── helm upgrade --install --atomic --wait  │
│  └── rollout summary (history + pods + hint) │
└──────┬───────────────────────────────────────┘
       │
       ▼
┌─────────────┐
│   smoke     │  scripts/smoke-prod.mjs against vars.SMOKE_URL
└─────────────┘
```

### Why `--atomic + --wait + --timeout`

- **`--atomic`**: if the upgrade fails (any resource fails to become ready before timeout), Helm automatically rolls back to the previous revision. No half-applied state. The workflow itself fails the job, surfacing the issue at the right boundary.
- **`--wait`**: blocks until all Deployments / StatefulSets / DaemonSets reach Ready and the rollout status is complete. Without this, the smoke job races the new pods.
- **`--timeout 10m`**: caps the wait so a stuck rollout fails the job rather than burning runner minutes indefinitely.

The trio together gives a synchronous deploy-with-failure-detection that's cleaner than the legacy SSH path's "tag image, run compose, hope it boots, smoke test, hope smoke catches anything wrong."

### Pre-flight gates inside `deploy`

Before the real `helm upgrade --install`, the workflow runs:

1. `helm lint <chart>` — catches structural breakage (missing required fields, invalid YAML, etc.) without touching the cluster.
2. `helm upgrade --install --dry-run --debug` — catches values breakage that lint can't see (e.g. a values file referencing a removed key, a nil dereference deep in a template). Renders the full manifest set against the live cluster's API server but doesn't apply.

Both fail-fast paths surface deploy errors **before** the cluster is touched. Critical for prod deploys.

## Decisions

- **Workflow stays `workflow_dispatch`-only.** The auto-deploy-on-CI-pass `workflow_run` trigger has been commented out since before OI-2 (the deploy host on the VM was Watchtower-managed). Re-enabling auto-deploy is a separate decision once the staging EKS environment is fully live and the smoke SLO is validated.

- **Image build stays in the deploy workflow.** Could have been split out to a separate "release" workflow consuming an existing GHCR image, but combining build+deploy in one workflow:
  - Lets operators deploy any ref (branch, tag, SHA) without pre-publishing the image
  - Keeps the audit trail (build → deploy → smoke) in a single workflow run
  - Build cache (`type=gha`) makes re-builds of the same ref nearly free
  
  The cost is ~5 min per deploy for a clean build; acceptable.

- **OIDC, not stored kubeconfig.** Storing a kubeconfig as a GH secret leaks the credential to anyone with workflow-edit access. OIDC short-lived STS credentials are scoped per-job + per-environment, with the IAM trust policy pinning to the specific repo + environment. Same model as terraform.yml.

- **`aws eks update-kubeconfig` over a kubeconfig artifact.** EKS issues IAM-authenticated kubeconfigs via the AWS CLI. The kubeconfig itself is short-lived (the auth plugin re-issues tokens on each kubectl call). No long-lived credential lands on the runner.

- **Per-env GitHub Environment for both deploy AND smoke.** Required-reviewers protection on `production` triggers the manual gate on the deploy job. The smoke job re-references the same environment so it can read `vars.SMOKE_URL` (which is env-scoped). Both jobs use the same OIDC role.

- **`--atomic` over manual rollback on failure.** Manual rollback would mean: deploy fails → workflow exits → operator notices → operator runs `helm rollback`. With `--atomic`, the rollback is automatic and immediate; the workflow surface just shows "deploy failed, rolled back". Cleaner failure mode.

- **`helm upgrade --install` (not `helm install` + `helm upgrade`).** The `--install` flag on `upgrade` makes the command idempotent — works for both first-time install (creates the release) and subsequent upgrades. One command path for both lifecycles.

- **Naming convention `inflect-<env>`** for both Helm release name AND K8s namespace. Matches the per-env values file naming (`values-staging.yaml` / `values-production.yaml`). Allows operators to run `helm list -A` and immediately read off which env each release belongs to. Locked by the structural ratchet.

- **Smoke retries / timeout knobs preserved verbatim** from the legacy workflow: `SMOKE_RETRIES=5`, `SMOKE_RETRY_DELAY=10000`, `SMOKE_TIMEOUT_MS=15000`. The smoke script's logic didn't change; only the URL source did (env var → GH Environment variable).

- **Helm chart `--history-max 10`.** Default is 10 in modern Helm but explicit-in-workflow makes the rollback window deterministic. 10 revisions is enough for ~2 weeks of daily deploys; bump if release cadence is higher.

- **Rollback documentation deliberately covers the migration-on-rollback footgun.** Helm rollback does NOT re-run pre-install/pre-upgrade hooks. So rolling the app image back to a revision that pre-dates a schema migration leaves the OLD app code reading the NEW schema — broken. The doc spells out the **expand-and-contract** discipline (forward-compatible schema changes only, two-PR lifecycle for column drops) as the structural mitigation. Calling out the constraint explicitly in the operator runbook prevents the surprise.

- **Older platform sections (Compose, Railway, Fly.io) preserved.** OI-2's K8s migration is the new primary path, but self-hosted scenarios still work. Removing the older sections would force smaller-scale deployments onto the EKS path unnecessarily. The deprecation banner makes the priority clear without breaking the alternates.
